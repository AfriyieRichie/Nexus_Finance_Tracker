Purpose & Objectives

This document guides the whole team through a structured User Acceptance Test (UAT) of Nexus Finance Tracker. The goal is to confirm that every feature works correctly with realistic data, that the accounting is sound (double-entry, balanced statements, correct GL postings), and that controls (approvals, segregation of duties, audit trail) behave as designed — before the system is relied upon for live operations.

Objectives

  • Exercise every module with real transactions, end to end.
  • Verify each posting hits the correct GL accounts and the books stay balanced.
  • Confirm approval workflows, segregation of duties and the audit trail enforce proper controls.
  • Validate that reports (Trial Balance, Balance Sheet, P&L, Cash Flow, Changes in Equity) reconcile to one another.
  • Capture any defects in a single log for triage and fixing.
ⓘ
Work through the phases in order. Accounting data has dependencies — you cannot post a journal before the Chart of Accounts and an open period exist, and you cannot test a period close before transactions are posted.
2

Scope

In Scope

All functional modules: Chart of Accounts, Accounting Periods, Journals & General Ledger, Accounts Receivable, Accounts Payable, Inventory, Fixed Assets, Bank Reconciliation, Payroll, Budgets & Cost Centres, Tax & Currency (VAT, FX revaluation), Approval Workflows, Financial Reports, Audit Trail, Dashboard, and User Management.

Out of Scope

  • Load / performance / stress testing.
  • Penetration & security testing (handled separately).
  • Third-party integrations / data migration from legacy systems.

Appendix — Notes & Pre-flight

A.1 Migration pre-flight (Org Admin, before Day 1)

Confirm these were applied in the Neon SQL editor. The enhancement migrations are required for: inventory PPV, NRV write-down, payroll lock, asset revaluation surplus & impairment reversal, budget description.

  • fixed_assets_enhancements_migration.sql — revaluation surplus, retained-earnings account, impairment-reversals table
  • inventory_enhancements_migration.sql — purchase price variance account
  • budgets_enhancements_migration.sql — budget description
  • payroll_enhancements_migration.sql — payroll locked_by / locked_at
  • (plus the base migrations: fixed assets, payroll, approvals, tax, AP, audit, user management)

Quick check query (run in Neon) — all four should return a row:

SELECT 'inv_ppv'      AS check, count(*) FROM information_schema.columns WHERE table_name='inventory_items' AND column_name='purchase_price_variance_account_id'
UNION ALL SELECT 'payroll_lock',  count(*) FROM information_schema.columns WHERE table_name='payroll_runs'    AND column_name='locked_at'
UNION ALL SELECT 'asset_surplus', count(*) FROM information_schema.columns WHERE table_name='fixed_assets'     AND column_name='revaluation_surplus_remaining'
UNION ALL SELECT 'impair_rev_tbl',count(*) FROM information_schema.tables  WHERE table_name='asset_impairment_reversals';

A.2 Testing tips

  • Segregation of duties: use different logins for create vs approve vs pay. The system enforces it and will block same-person actions.
  • GL verification: after each posting, open the General Ledger or Trial Balance and confirm the exact accounts and amounts.
  • One change at a time: if a result looks wrong, note the test Ref and exact values in the Defect Log so it can be reproduced.
  • Dates matter: post transactions inside an OPEN period; FX revaluation uses the period-closing rate on/before the chosen date.
  • Don't lock too early: download payroll payment files and finish reconciliations before locking runs/periods.

A.3 What "good" looks like

By the end, the team should be able to demonstrate: a full transaction cycle from invoice to cash, a payroll run posted and locked, depreciation and asset events posted, a reconciled bank statement, a balanced trial balance, four financial statements that reconcile to each other, and an audit trail that captured every action — all under enforced approval controls.
